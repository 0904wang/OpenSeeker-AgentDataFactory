## Analysis

The pipeline sustained very high request concurrency and received 199 successful teacher generations. The final request became a tail-latency outlier and eventually returned invalid or empty JSON, producing `teacher_backend_error=Expecting value: line 1 column 1 (char 0)`. Because the current implementation waits for every future before verification/export, a single slow request can make the run appear stuck even though raw streaming is working.

The practical issue is not GPU or CPU pressure; it is tail latency and lack of a hard per-sample wall-clock budget around teacher calls. For resume-quality data factory work, this is an important engineering finding: high-throughput generation needs resumability, per-batch checkpointing, partial export/recovery, and clearer progress timestamps.

## Next Steps

- Treat this as a successful 200-sample seed expansion smoke run, but do not scale directly to 1k/20k with the current single-shot runner.
- Add a robust batched generation mode before scaling: smaller batches, append-only outputs, per-batch summaries, resume from completed sample ids, and timestamped progress logs.
- Prefer concurrency 20-50 until the runner has batch-level recovery or a hard timeout around teacher futures.
- Use this result as evidence that the verifier/fallback path works: final accepted rate was 100%, teacher success rate was 99.5%, and fallback rate was 0.5%.
